# DocMind 文档 Agent 路线图

## 1. 目标

把 DocMind 从“能跑的文档问答系统”继续做成“真正好用的文档 Agent”。

这里的“好用”具体指：

1. 找得到真正相关的文档证据
2. 回答结构稳定，不空泛
3. 能处理追问、指代和版本问题
4. 证据不足时会明确说明，而不是硬编
5. 质量能通过真实样本和运行指标持续验证

## 2. 当前基础

已经具备：

- Runtime V2 主链路
- ES / Milvus / Neo4j 检索底座
- 上传、入库、状态和原文访问链路
- SSE 对话和历史会话
- 安全、审计、评估、训练编排入口

所以当前路线图的重点不再是“补页面”和“接脚手架”，而是增强文档理解与回答质量。

## 3. 当前主要短板

### 回答像系统，不像助手

- 有时能找到证据，但回答结构不够稳定
- 结论、依据、引用、风险提示还不够统一

### 追问能力偏弱

- 对“这个什么时候生效”“和上一版比有什么变化”这类问题还不够稳
- 版本、制度名称、时间范围的澄清仍需加强

### 检索策略还不够任务化

- 不同类型问题仍共享较多检索路径
- 摘要、制度问答、关系查询、对比分析应有不同召回策略

### 质量评估还要更贴近真实任务

- 健康检查主要证明链路可用
- 还需要更有代表性的文档问答样本和任务型评测

## 4. 路线图分阶段

### P0：中文质量与回退一致性

目标：彻底清理中文乱码、提示词污染和用户可见异常文案。

重点：

- 查询改写提示词
- 聊天页文案
- 安全拦截文案
- 规则词和别名词表

### P1：任务感知检索

目标：让系统先更会“找”。

重点：

- 按问答、摘要、比较、关系、统计等类型差异化检索
- 增强版本、标题、章节、部门等权重
- 显式处理无证据和弱证据

### P2：结构化回答

目标：让系统更会“答”。

重点：

- 制度问答：结论 / 依据 / 引用
- 摘要：主题 / 要点 / 风险 / 待确认项
- 对比：相同点 / 差异点 / 影响
- 合规：要求 / 风险 / 建议动作

### P3：多轮追问

目标：让文档会话不再失忆。

重点：

- 指代消解
- 上下文继承
- 版本与生效时间澄清
- 历史引用复用

### P4：质量门禁

目标：让“好不好用”有证据，而不是凭感觉。

重点：

- 建立更真实的文档问答评测集
- 跟踪命中率、空回答率、重复引用率、澄清率
- 将评估结果与交付预检衔接

## 5. 明确不优先做的事

- 不优先把它做成通用代码代理
- 不优先追求 UI 花哨改版
- 不把“换更大模型”当成首要解法
- 不为了健康分数好看而降低真实质量门槛

## 6. 当前路线判断

当前系统已经过了“从零到一”的阶段。后续工作的价值主要来自质量优化密度，而不是继续堆框架名词。
