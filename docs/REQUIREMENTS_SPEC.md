# DocMind 需求规格说明书

## 1. 文档说明

### 1.1 编写目的

本文档用于明确 DocMind 企业文档管理与智能问答平台的建设目标、业务边界、用户角色、功能需求、非功能需求与验收基线，为课程答辩、阶段交付、后续迭代开发和测试验证提供统一依据。

### 1.2 适用范围

本文档适用于当前仓库 `documentmanagement_agent` 的 Web 端、后端服务、异步处理链路、检索链路和管理后台。本文档以当前代码实现、接口定义和部署脚本为依据整理，不把外部公共网页材料视为原始官方需求。

### 1.3 术语说明

- DocMind：本项目的企业文档管理与智能问答平台名称。
- 租户：平台中的组织隔离单元，不同租户的数据默认隔离。
- 文档入库：文档上传后进入解析、切块、索引和状态回写流程。
- 混合检索：关键词检索、向量检索、图谱检索等多种检索策略的组合。
- SSE：服务端事件流，用于问答过程中的流式状态与答案回传。
- Fallback：在真实 AI 或外部依赖不可用时系统使用的降级实现。

## 2. 项目背景与建设目标

### 2.1 项目背景

企业内部制度、流程、财务、合同、采购、科研和行政类文档分散在不同系统与文件中，存在以下共性问题：

- 文档来源多、格式复杂，人工检索效率低。
- 相同问题重复咨询，缺少可追溯的统一问答入口。
- 上传、解析、索引、检索、问答和安全审计之间缺少闭环。
- 管理人员难以及时掌握任务状态、检索质量和运行健康度。

### 2.2 建设目标

系统建设目标如下：

1. 提供统一的企业文档上传、存储、处理、检索和问答平台。
2. 在不依赖真实大模型也能运行的前提下，完成核心流程闭环。
3. 保留真实 LLM、Embedding、Reranker、OCR、评估能力的稳定接入边界。
4. 建立认证、权限、租户隔离、安全审计、运行可观测等工程基线。
5. 为后续企业领域模型训练、评估与上线提供数据和系统基础。

## 3. 业务范围与边界

### 3.1 本期纳入范围

- 用户登录、刷新令牌、当前用户信息、邀请注册、邮箱验证、密码重置。
- 文档上传、分片上传、列表查询、处理状态查询、事件查询、重试和删除。
- 关键词检索、向量检索、图谱检索、混合检索。
- 智能问答、SSE 流式事件回传、会话历史、反馈提交。
- 管理后台总览、任务管线、安全审计、系统后端状态、检索一致性、运行时指标、评估入口。
- Docker 本地部署、基础脚本、健康检查、CI 基线。

### 3.2 本期不承诺完成的范围

- 金融级或强监管级别的最终安全策略闭环。
- 完整生产级 OCR、版面分析和复杂表格结构抽取能力。
- 完整真实灰度流量调度和多模型生产发布体系。
- 针对正式业务部门的定制流程审批或外部 ERP 深度集成。

## 4. 用户角色与权限需求

### 4.1 角色划分

- 普通用户：上传文档、查看本人有权访问的文档、发起检索和问答、查看历史。
- 管理员：具备普通用户能力，并可访问管理后台、用户与系统运维类接口、失败任务治理、平台状态与评估视图。

### 4.2 权限需求

1. 系统必须基于 JWT 进行会话认证。
2. 系统必须基于角色控制管理接口访问权限。
3. 系统必须基于租户隔离用户、文档、会话、审计和运行时数据。
4. 文档访问必须考虑部门和访问级别等控制信息。

## 5. 功能需求

### 5.1 认证与账号管理

系统应支持以下能力：

1. 用户登录并返回访问令牌与刷新令牌。
2. 已登录用户获取当前账号信息。
3. 使用刷新令牌更新访问令牌。
4. 管理员发起邀请注册，并可查询、重发、撤销邀请。
5. 发送邮箱验证码并完成邮箱验证。
6. 发起密码重置与确认重置。
7. 默认关闭公开注册，只允许按策略开放。

### 5.2 文档管理

系统应支持以下能力：

1. 上传 PDF、DOCX、XLSX、CSV、PNG、JPEG 等格式文档。
2. 校验文件类型、大小限制和基础安全扫描结果。
3. 支持普通上传与大文件分片上传。
4. 记录文档基础元数据、所属租户、部门、访问级别和上传人。
5. 提供文档分页列表，支持部门和状态筛选。
6. 提供处理状态查询，包括进度、错误信息、尝试次数和更新时间。
7. 提供最近处理事件查询，便于用户和管理员追踪问题。
8. 提供失败重试与文档删除能力。

### 5.3 文档处理与入库

系统应支持以下能力：

1. 文档上传后异步进入解析、切块、向量化和索引流程。
2. 系统应回写处理进度和阶段事件。
3. 文档处理结果至少支持 `queued`、`parsing`、`chunking`、`indexing`、`ready`、`partial_failed`、`failed` 等可观测状态。
4. 在部分外部能力不可用时，系统仍可通过 fallback 维持主链路可运行。

### 5.4 检索能力

系统应支持以下能力：

1. 关键词检索。
2. 向量检索。
3. 图谱检索。
4. 混合检索。
5. 直接检索接口，供前端或调试场景绕过 Agent 使用。
6. 对检索健康度、成功率、超时率和一致性提供可观测能力。

### 5.5 智能问答与会话

系统应支持以下能力：

1. 用户输入问题后发起 SSE 流式问答。
2. 流式事件应至少覆盖思考、检索、阅读、工具调用、内容流式输出、完成、错误等过程状态。
3. 系统应支持会话历史查询。
4. 系统应支持用户对回答提交评分和修正反馈。
5. 运行时应支持事件 replay、trace 追踪和断点续传能力。

### 5.6 管理后台

系统应支持以下管理能力：

1. 用户列表查询。
2. 平台总览。
3. 文档处理任务列表、失败概览、单任务重试、批量重试。
4. 安全审计事件查询。
5. 后端组件状态查看，包括检索、缓存、安全和 AI 相关依赖。
6. 检索指标、检索一致性、平台就绪度查看。
7. 离线评估运行、最新评估报告查询、运行时评估指标查询与导出。
8. 运行时任务、工具决策、决策汇总、事件 replay、checkpoint 摘要查询。

## 6. 非功能需求

### 6.1 可用性

1. 在真实 AI 服务不可用时，核心上传、检索、问答主链路仍应可运行。
2. 用户可见关键状态必须可观测，不能只在后台静默失败。

### 6.2 性能

1. 系统应支持基本并发下的上传、检索和流式问答。
2. 系统应能记录 TTFT、完成耗时、回退率、拒绝率等运行指标。
3. 系统应保留基础压测与预检脚本。

### 6.3 安全

1. 系统应具备认证、角色鉴权、租户隔离和审计基线。
2. 上传必须经过基础安全扫描。
3. 对敏感输入输出应具备可扩展的安全守卫接口。
4. 系统必须避免跨租户数据泄露。

### 6.4 可维护性

1. 系统应采用前后端分层结构和模块化接口。
2. 系统应保留 AI 接入占位边界，避免临时改坏前后端契约。
3. 系统应具备测试、脚本和文档，便于后续维护。

### 6.5 可扩展性

1. 后续应可替换或增强 LLM、Embedding、Reranker、OCR 和评估实现。
2. 后续应可扩展到更多租户、更多文档类型和更复杂的企业语料。

## 7. 数据需求

### 7.1 业务数据

- 用户、角色、租户、邀请、验证码、密码重置记录。
- 文档元数据、处理状态、chunk 元数据、引用、反馈、审计事件。
- 会话历史、运行时事件、评估报告。

### 7.2 参考语料来源

当前仓库内已有两类公共语料目录，可用于冷启动领域样本和测试数据准备：

- `datasets/public_enterprise_docs/`
- `datasets/swu_public_docs/`

这些资料仅作为公开制度文档样例与领域背景，不视为本项目的正式原始需求说明。

## 8. 外部接口需求

### 8.1 前后端接口

系统前后端通过 REST API 与 SSE 通信，主要包括：

- 认证接口
- 文档接口
- 检索接口
- 问答接口
- 管理后台接口

### 8.2 外部基础设施依赖

- PostgreSQL
- Redis
- MinIO
- Elasticsearch
- Milvus
- Neo4j
- Docker Compose

### 8.3 可选 AI 与安全依赖

- Ollama 或 OpenAI-compatible LLM 接口
- Embedding 服务
- Reranker 服务
- OCR 依赖
- Guardrails / Presidio / Ragas sidecar

## 9. 验收要求

本期验收至少应覆盖以下内容：

1. 后端健康检查正常。
2. 登录、当前用户、刷新令牌可用。
3. 文档上传、状态流转、事件查询、重试与删除可用。
4. 关键词检索和至少一种混合问答主链路可用。
5. SSE 前端可正确接收并展示标准事件流。
6. 管理后台能查看任务、系统状态、检索一致性和评估结果。
7. 前后端构建和基础测试通过。
8. 主要用户可见中文文案无明显乱码。

## 10. 后续迭代建议

1. 将当前需求文档进一步细化为页面级和接口级用例说明。
2. 将领域公共语料整理成正式的测试数据集和演示数据集。
3. 按照当前 AI 占位边界逐步替换为真实服务，并补足真实模式下的测试门禁。
